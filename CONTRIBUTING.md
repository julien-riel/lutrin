# Contributing to Lutrin

Thank you for your interest. This document says how the project works — what
is expected of a contribution, and above all what is not.

By submitting a contribution, you agree that it be distributed under the
project's MIT licence (see `LICENSE`).

## Getting started

```sh
git clone https://github.com/julien-riel/lutrin.git && cd lutrin
npm install
npm test                    # all three packages (core, VS Code, Obsidian)
npm run test:core           # the engine alone, when that is what you touch
npm run typecheck           # both editor hosts (tsc --noEmit)
npm run lint                # biome — BLOCKING in CI
node packages/core/src/cli.mjs build examples/demo.deck.md -o /tmp/demo.pptx
```

`npm run lint` (`biome check .`) is the CI `format` job, and it is
**blocking**: a PR that does not pass is red before it is read. Run it before
you push. `npm run fmt` (`biome format --write .`) rewrites what is only
formatting; what the formatter cannot repair on its own is fixed by hand, or
by `npx biome check --write <files>`. The rules the project does not endorse
are disabled in `biome.jsonc`, each with its reason — so you only have to
fix what your PR introduces.

Node ≥ 22 is required (`engines.node` of the four `package.json` files,
locked down by `manifests.test.mjs`). The purely technical constraint would
be a little lower — the glob of `node --test "<pattern>"` only exists from
version 21 onwards, and pptxgenjs ships ESM that only the syntax detection
introduced in 20.19 knows how to load — but the 20 line is reaching end of
life: the project does not support it. That is a policy choice, not a limit
of the code.

No build tool for the compiler: the sources run as they are (ESM). Only the
extensions go through TypeScript and esbuild.

## The project contract

**The author describes intent and content; the engine decides the layout.**
Markdown is not the final format, it is a DSL. A proposal that would ask the
author to position elements (coordinates, sizes, explicit columns) goes
against this contract and will be refused, however well implemented.

Three corollaries:

- **Output parity** — the `.pptx` and the HTML are born of the same scene. A
  feature that exists on one side only must degrade cleanly on the other, not
  diverge silently.
- **The document always wins** — the deck's frontmatter takes precedence over
  any project, user or editor configuration.
- **`build` does not deliver a deck that is in error** — as soon as one
  diagnostic of severity `error` remains, the command exits with code 1 and
  writes no file; only `--force` overrides it. A change that would make an
  `error` harmless must therefore change its severity, not work around the
  guard. Corollary of the corollary: a kit that comes from an implicit
  default (project, user, host) and is not found stays a mere warning —
  without which a stale configuration would block projects that asked for
  nothing.

## Code expectations

- **English** for comments, diagnostic messages and documentation.
- **Comments explain the WHY**, not the what. A comment that paraphrases the
  next line will be removed; a comment that explains why an obvious solution
  was ruled out will be kept.
- **No new dependency without discussion.** The test harness is `node:test`,
  without exception.
- **`src/deck/` is the core**: it knows no output format and imports no
  backend library. `boundary.test.mjs` checks this.

## Tests

Every bug fix comes with a test that **fails before** the fix. Check it:
comment out your fix, the test must go red. A test that passes either way
tests nothing — that happened in this repository, and it is how the
path-traversal tests came to be rewritten.

`examples/demo.deck.md` is the renderer coverage fixture: **every new block
type must be added to it**. The goldens are regenerated by
`UPDATE_GOLDEN=1 npm test` — re-read the diff before committing it, it must
correspond exactly to the change you intended.

## Kits

A **kit** is a directory (or a `.deckkit` archive) carrying a `kit.json`, a
`theme.json` and optionally layouts, fonts and logos. See
`examples/kit-slate/` for a complete, rights-free example.

A kit contains **data only, never code**: that is what makes installing it
from a URL defensible, and `lutrin kit install` refuses any executable entry.
Do not propose loosening this rule.

The project distributes no branded kit. An organization's brand lives in its
own repository, under the control of whoever owns the trademark.

An organization building its own VS Code host can embed its kit in the VSIX
at build time: point the `LUTRIN_BRAND_KIT` environment variable at a kit
directory or a `.deckkit` archive — or write that path into a `.brandkit`
file at the repository root (untracked) — before `npm run vsix -w
lutrin-vscode`. The kit is copied into the package under the name its
`kit.json` manifest declares. Its absence is never an error (a public clone
builds on the generic theme); a kit that is asked for but not found fails
the build rather than shipping a package silently stripped of its brand.

## Reporting a problem

A good issue contains the minimal Markdown that reproduces the problem, the
command that was run, what was expected and what happened.
`lutrin validate <deck.md> --json` gives the diagnostics in a machine-readable
format.

For a security vulnerability — especially around `kit install`, which writes
bytes that came from outside onto the disk — **do not open a public issue**:
go through **GitHub Security Advisories** (the repository's *Security* tab →
*Report a vulnerability*). The procedure and the target turnaround times are
in [SECURITY.md](SECURITY.md).
